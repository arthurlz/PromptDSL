# promptc — a Prompt DSL compiler (Design)

**Date:** 2026-06-14
**Status:** Design — awaiting review

## Context

`requirements.md` captured an AI brainstorm of five possible project directions
for someone who enjoys OCaml/Racket, PL, DSLs, and AI. The chosen direction is
**#1, a Prompt DSL**: a small language that describes an AI agent's goal and
steps, and *compiles* that description into prompts/payloads for LLM providers.

The motivation: "in the AI 2.0 era, precisely describing intent matters more and
more." Rather than hand-writing brittle prompt strings, you write a structured,
checkable `.prompt` source and let a compiler produce provider-specific output.
This plays directly to PL strengths (lexer, parser, AST, semantic analysis, IR,
backends) while producing something genuinely useful.

**Intended outcome of this MVP:** a working `promptc` CLI that takes a
`.prompt` file and emits, for OpenAI, *both* a human-readable prompt and a
runnable Chat Completions JSON payload — with friendly, span-based diagnostics.

## Decisions (locked)

| Decision | Choice |
| --- | --- |
| Project direction | Prompt DSL (requirements.md #1) |
| Implementation language | **OCaml** (despite the `rust/` parent folder) |
| Compile output | **Both** human-readable prose **and** structured JSON request |
| MVP target providers | **OpenAI only**, behind a backend interface that admits Anthropic/Gemini later |
| Static analysis | One **basic semantic pass** (no full type system) |
| `output` spec | Format keyword **with an optional typed schema** |
| Parser strategy | **Menhir + ocamllex** |
| Source extension | `.prompt` (the doc's `.ps` collides with PostScript) |

## The language (MVP surface)

```
agent "researcher" {
  goal "分析 TSLA 财报"

  step { search "TSLA earnings" }
  step { summarize }

  output json {
    ticker:  string
    rating:  enum("buy", "hold", "sell")
    summary: string
  }
}
```

Constructs:

- `agent "<name>" { ... }` — single top-level block (MVP: exactly one agent per file).
- `goal "<text>"` — the objective; becomes the core of the system prompt. Required.
- `step { <action> ["<arg>"] }` — ordered steps. Zero or more.
  - **Built-in action vocabulary** (closed set for the MVP, easy to extend):
    `search`, `summarize`, `extract`, `translate`, `classify`.
  - `instruct "<text>"` — generic escape hatch for an arbitrary instruction.
  - Each action takes an optional single string argument (e.g. `search "TSLA earnings"`,
    `translate "ja"`). Unknown actions are a semantic error with a "did you mean…" hint.
- `output <format>` — optional; defaults to `text` if omitted.
  - `output markdown` / `output text` / `output json`
  - `output json { <fields> }` — an optional typed schema. Field types for the MVP:
    `string`, `int`, `bool`, `enum("a","b",...)`, and `list<T>`. Each field is
    required unless suffixed with `?` (e.g. `note?: string`).

UTF-8 string literals (Chinese, etc.) are supported; identifiers and keywords are ASCII.

## Architecture

Classic multi-pass compiler. Each stage is a separate, independently testable module.

```
source.prompt
  → Lexer (ocamllex)        → tokens
  → Parser (Menhir)         → raw AST (every node carries a source span)
  → Sema (one pass)         → checked AST   (resolve refs, validate actions/formats,
                                              dedupe fields, accumulate diagnostics)
  → Lower                   → IR            (provider-agnostic prompt model)
  → Backends ─┬─ Prose      → prompt string
              └─ OpenAI     → JSON request payload (messages + response_format)
```

**IR** is the provider-agnostic seam that makes "both outputs" and "more providers
later" cheap. It holds: agent name, system objective (from `goal`), an ordered list
of normalized instructions (from `step`s), and an optional output schema.

### Module layout (dune project)

```
dune-project
bin/
  main.ml                 # CLI (cmdliner)
lib/
  location.ml             # positions, spans, span-merging
  error.ml                # diagnostics: kind + span + message + optional hint; pretty-printer
  lexer.mll               # ocamllex token rules
  tokens.ml               # token type shared by lexer/parser (or generated by Menhir)
  parser.mly              # Menhir grammar → raw AST
  ast.ml                  # raw AST types (with Location.t)
  sema.ml                 # semantic pass → checked AST or diagnostics
  ir.ml                   # provider-agnostic IR types
  lower.ml                # checked AST → IR
  backend_prose.ml        # IR → human-readable prompt
  backend_openai.ml       # IR → OpenAI JSON (yojson)
  compile.ml              # orchestrates the pipeline; returns outputs or diagnostics
test/
  ...                     # alcotest unit tests + cram golden tests
examples/
  researcher.prompt
```

**Dependencies:** `dune`, `menhir`, `ocamllex` (bundled), `yojson`, `cmdliner`,
`alcotest`. Optional: `ppx_deriving.show` for AST/IR debug printing.

## CLI

- `promptc compile <file.prompt> [--emit prose|json|both] [--target openai] [-o <dir>]`
  - default `--emit prose`, default (and only) `--target openai`.
  - `both` writes/prints both artifacts; `-o` writes files, otherwise stdout.
- `promptc check <file.prompt>` — run lexer→parser→sema only; print diagnostics, exit non-zero on error.

## Compilation semantics (what the backends produce)

- **Prose backend:** a system+user style prompt. The `goal` frames the role/objective;
  steps render as an ordered, numbered instruction list; an `output json {schema}`
  appends "Return ONLY JSON matching this schema:" followed by the schema rendered
  readably.
- **OpenAI backend:** a Chat Completions request object:
  - `messages`: a `system` message (role/goal + the rendered steps) and a `user`
    message placeholder.
  - `response_format`: when `output json {schema}` is present, a
    `json_schema` built from the typed schema; otherwise omitted (or `text`).
  - `model`: a sensible default (e.g. `gpt-4o-mini`), overridable later.

The MVP **does not execute** anything (no network calls, no tool running). Steps
like `search` compile to *instructions*, not actions. Execution is explicitly
future work (requirements.md's "future: output workflow").

## Error handling

- Every diagnostic carries a source span and prints as:
  `file.prompt:3:5: error: unknown action 'searchh' (did you mean 'search'?)`
- The semantic pass **accumulates** errors (reports as many as it safely can) rather
  than stopping at the first.
- Parser errors use Menhir's error mechanism to report the expected tokens.

## Testing

- **alcotest unit tests:** lexer (token streams), parser (AST shape on small inputs),
  sema (each error category fires; valid input passes), lower (AST→IR mapping).
- **cram golden tests (the backbone):** a corpus of `.prompt` files run through
  `promptc compile --emit both`, with expected prose + JSON checked in. Adding a
  language feature = add an example + its golden output. Error-case `.prompt`
  files assert on the exact diagnostics.

## MVP scope

**In scope:** the language surface above; lexer→parser→sema→lower→prose+OpenAI
backends; `compile`/`check` CLI; diagnostics with spans; example corpus; tests.

**Explicitly out of scope (future):**
- Anthropic and Gemini backends (the IR/backend seam is designed for them, but they
  are not built now).
- Executing steps / tool calls / running a workflow.
- A full type system (`prompt<In,Out>`, input schemas, inference) — direction #3.
- Multiple agents per file, variables/templating, conditionals, imports.
- Formatter (`promptc fmt`), LSP, WASM playground.

## Suggested milestones (for the implementation plan)

1. **Skeleton:** dune project, `location.ml`, `error.ml`, CLI stub, first cram test harness.
2. **Frontend:** `lexer.mll` + `parser.mly` + `ast.ml`; parse `examples/researcher.prompt` to an AST, verified by a parser unit test.
3. **Sema:** validate actions/formats, dedupe schema fields, accumulate diagnostics; `promptc check` works with good error messages.
4. **IR + lowering:** `ir.ml`, `lower.ml`.
5. **Backends:** `backend_prose.ml`, then `backend_openai.ml` (yojson, `response_format`).
6. **Polish:** `--emit both`, golden-test corpus (happy + error paths), README with a worked example.
